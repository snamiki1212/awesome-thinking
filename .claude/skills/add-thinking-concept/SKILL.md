# 思考概念の追加ワークフロー

このリポジトリに新しい思考概念を1つ追加するための手順をガイドする。
方針の正本は `CLAUDE.md`、タイプ定義の正本は `docs/taxonomy.md`。迷ったら必ずそちらを参照する。

ゴールは「テンプレートに沿った詳細ファイル1枚」＋「トップ README への1行追加と `index.md` の再生成」を、相互リンクと言語方針を守って完成させること。

## 大前提（言語とスラッグ）

- 本文・見出し・説明・コミットメッセージは**すべて日本語**で書く。
- ファイル名・ディレクトリ名（スラッグ）だけは**英語の kebab-case**（例: `first-principles`）。
- スラッグに対応する日本語名は、フロントマターの `title` と本文 H1 タイトルに必ず併記する。

## 手順

### 0. 収録可否を判定する（受け入れ要件 / admission gate）

タイプを考える**前に**、まず「そもそも思考ツールとして収録対象か」を判定する。正本は [`docs/taxonomy.md`「収録の前提」](../../../docs/taxonomy.md)。

**収録するのは、次を *すべて* 満たすもの：**

1. 「どう考えるか・どう捉えるか」を与える**思考の道具**である（対象知識・事実そのものではない）。
2. 特定の業界・製品・課題に閉じず**汎用的に転用できる**。
3. 4タイプ（メンタルモデル／フレームワーク／思考法／思考術）の**いずれかに自然に収まる**。
4. 1ファイルで**定義でき**、既存項目の単なる別名ではない。

**次のいずれかに当たるものは reject（収録しない）：**

- ドメインの専門知識・事実・データそのもの（思考の対象であって道具ではない）
- 思考の進め方・見方を与えない単なる用語・概念
- 製品・サービス・ツール・特定の会議体などの固有名
- 心構え・感情・価値観だけで、思考の操作や視点を伴わないもの
- 既存項目の単なる別名・言い換え（→ 既存項目の `aliases` に足すことを提案する）
- 出典も実体も確認できず、内容を定義できないもの

判定に迷う**境界事例**は reject せず、最も主要な用途で4タイプに寄せて収録してよい（手順1へ進む）。

**reject すると判断したとき：**

1. ユーザーに「なぜ収録対象外か」を上記のどれに当たるかで端的に伝える。代替（別名として既存項目に追記する等）があれば提案する。
2. [`docs/out-of-scope.md`](../../../docs/out-of-scope.md) の一覧表に1行追加する（`日付 / 項目（提案された呼称）/ 提案された区分 / 却下理由`）。
3. 詳細ファイルやインデックスは**作らない**。ここで手順を終える。
4. 受け入れ要件そのものを変える・新タイプを足す判断をした場合のみ、別途 ADR を起こす（手順6）。

収録対象と判断したら、手順1へ進む。

### 1. タイプを判定する

`docs/taxonomy.md` を読み、4タイプのどれに属するか決める。判断は次の順に問うと早い：

1. **手順や埋める枠があるか？** → `thinking-frameworks/`（思考フレームワーク）
2. **世界の見方・前提として効くか（手順はない）？** → `thinking-mental-models/`（メンタルモデル）
3. **思考全体の方向性・姿勢を決める大きな構えか？** → `thinking-methods/`（思考法）
4. **その場ですぐ打てる小さな技か？** → `thinking-skills/`（思考術）

複数にまたがる場合は最も主要な用途のディレクトリに置き、本文「関連」に他タイプとの関係を書き残す。
**分類のルール自体を変える・新タイプを足す判断は、項目追加ではなく ADR 案件**（後述）。

| タイプ | ディレクトリ | `type` 値 |
| --- | --- | --- |
| メンタルモデル | `thinking-mental-models/` | `mental-model` |
| 思考フレームワーク | `thinking-frameworks/` | `framework` |
| 思考法 | `thinking-methods/` | `thinking-method` |
| 思考術 | `thinking-skills/` | `thinking-skill` |

### 2. スラッグを決め、重複を確認する

英語 kebab-case のスラッグを決め、既存ファイルと衝突しないか確認する：

```
ls <タイプのディレクトリ>/
```

別名・英語名が異なるだけの既存項目がないか（重複収録の防止）も、各ディレクトリの `index.md`（一覧）でざっと確認する。

**命名のチェック（日本語タイトル・英語名・スラッグ・`aliases`）** — 経緯: [adr/20260608-prefer-established-general-naming](../../../adr/20260608-prefer-established-general-naming/README.md)

- **一般的・確立した呼称を優先する**: 特定の著者・一冊の本の造語ではなく、その分野で国際的・一般的に通用する呼称を選ぶ。出典が一冊に偏るときほど、より広く使われる呼称がないか先に調べる（例: 一著者の「不確実性の削減」より一般的な「不確実性マネジメント／Managing Uncertainty」）。
- **英語名の衝突を確認する**: 英語名・`aliases` が、別分野で確立した同名・類似名の概念と紛れないかを確認し、紛らわしければ避ける（例: `Uncertainty Reduction` は対人コミュニケーション論の別理論 "Uncertainty Reduction Theory" と取り違えやすい）。
- **内部の収まりで決めない**: 命名・タイプ・採否は概念自体の一般性・正確さ・出典で決める。「既存項目との対比が綺麗」「リポジトリに収まりがよい」といった内部都合を根拠にしない。既存項目との対比・補完は `関連`／`似ている用語との違い` 節で表現すれば足りる。

### 3. 詳細ファイルを作成する

`<タイプのディレクトリ>/<english-slug>.md` を、下記テンプレートで作る。
節は省略せず全部埋める（例外は任意節の `図解` のみ）。冗長にせず「1項目で読み切れる」分量に保つ。

```markdown
---
title: 日本語タイトル
slug: english-slug
type: mental-model | framework | thinking-method | thinking-skill
aliases: [別名, English Name]
description: "「一言でいうと」の内容を1文で"
timestamp: yyyy-mm-ddTHH:MM:SS+09:00   # 最終更新日時（ISO 8601）
tags: [意思決定, 問題解決]   # 目的・ドメインの参考用
---

# 日本語タイトル（English Name）

## 一言でいうと
（1〜2文）

## 定義
（何であるか）

## 図解
（任意の節。構造そのものが図的な概念のみ、mermaid の図を1つ置く。該当しなければ節ごと省略する）

## 使いどころ
（どんな場面で有効か）

## 使い方・手順
（具体的な進め方。箇条書きや番号付きで）

## 例
（具体例）

## 注意点・落とし穴
（誤用やよくある勘違い）

## 似ている用語との違い
（名前や効果が似ていて混同されやすい概念を挙げ、「どこが似ていて・どこが違うか」を一言で示す。混同の解消が目的）
- **似た用語（English Name）** — 似ている点 → 決定的な違い。
  例: 複利 vs レバレッジ … どちらも「効果を増幅する」点で似るが、複利は**時間をかけて積み上がる**もの、レバレッジは**他者の資源で一度に拡大する**もの。
- このリポジトリに該当項目があれば `[スラッグ](相対パス)` の Markdown リンクでリンクする（無くてもよい）。

## 関連
- [関連スラッグ](../<タイプのディレクトリ>/<関連スラッグ>.md)（日本語名）— ひとこと補足
```

フロントマターは OKF-AT（このリポジトリのナレッジ管理フォーマット）の必須フィールドを満たす必要がある。規約の正本は [`docs/okf-at/README.md`](../../../docs/okf-at/README.md)（CI で検査される。ローカル確認は `node scripts/lint-okf-at`）。

執筆スタイルの要点（`CLAUDE.md` 準拠）：
- 専門用語は初出時に短い説明を添える。
- 提唱者・書籍など出典が明確なら分かる範囲で触れる。ただし**不確かな出典を断定しない**。
- **出典は単一に依存しない**。可能なら複数・国際的な裏づけを併記し、特定著者の整理は「整理の一つ」として明示する（汎用の確立概念であるかのように断定しない）。経緯: [adr/20260608-prefer-established-general-naming](../../../adr/20260608-prefer-established-general-naming/README.md)。
- メンタルモデルは「手順」を持たないので、`使い方・手順` は「どう視点として使うか」に寄せる。
- `図解` は**任意の節**。概念の構造そのものが図的な場合（木構造・循環・2軸の行列・フィードバックループ・抽象と具体の往復など）に限り、mermaid の図を1つ置き、図の前に何を図示したかを1〜2文で述べる。本文の言い換えにしかならない図は置かず、節ごと省略する。経緯: [adr/20260713-optional-diagram-section](../../../adr/20260713-optional-diagram-section/README.md)
- `似ている用語との違い` と `関連` は役割が違う。前者は**混同しやすい別概念を切り分ける**節（リポジトリ未収録の用語でもよい）、後者は**一緒に使える／つながる項目を相互リンクする**節。該当する似た用語が思い当たらなければ「特になし」とだけ書いてよい。

### 4. インデックスを更新する（★ここが最も漏れやすい）

一覧は2種類あり、更新方法が違う。

1. **トップ README（手書き）** — ルートの `README.md` 内、全ジャンルをまとめた**単一の統合テーブル**（`| ジャンル | 項目 | ひとこと |`）に
   `| <ジャンルのラベル> | [日本語タイトル](./<dir>/<slug>.md) | ひとこと |` を1行追加する。
   - 行は同じジャンルがまとまるよう、該当ジャンルの既存行の隣に挿入する。
   - 「ジャンルのラベル」は既存行と同じ表記（絵文字つき。例: `🧠 メンタルモデル` / `🧩 思考フレームワーク` / `🌊 思考法` / `⚡ 思考術`）にする。
   - 「ひとこと」は詳細ファイルの `## 一言でいうと` を1行に凝縮したものにする。リンクの相対パス（`./<dir>/<slug>.md`）を確認する。
2. **各ディレクトリの `index.md`（自動生成）** — `node scripts/gen-okf-index` を実行して再生成する。手で編集しない。
   一覧の中身は詳細ファイルの frontmatter（`title` / `description`）から引かれる。

ジャンル別 README には一覧表を書かない（一覧は `index.md` に委ねる。経緯: [adr/20260713-readme-delegates-listing-to-index](../../../adr/20260713-readme-delegates-listing-to-index/README.md)）。

### 5. 相互リンクする

関連する既存項目があれば、双方の `## 関連` を `[スラッグ](相対パス)` の Markdown リンクで**相互に**つなぐ。
相対パスは参照元ファイルから見たパスにする（同じディレクトリなら `./<スラッグ>.md`、別タイプなら `../<タイプのディレクトリ>/<スラッグ>.md`）。
wikilink（`[[スラッグ]]`）は GitHub 上でリンクとして機能しないため使わない（正本: [`docs/okf-at/README.md` のリンク規約](../../../docs/okf-at/README.md)）。
新項目から既存項目へリンクを足したら、可能なら既存項目側からも新項目へ1行足す。

### 6. ADR が必要か判断する

単なる項目追加（既存ルールに沿うもの）や軽微な修正は ADR 不要。
ただし以下に当てはまるなら、別途 ADR を `adr/yyyymmdd-<name>/README.md` に起こす（`CLAUDE.md` のテンプレ・手順に従い、`node scripts/gen-okf-index` で一覧を再生成する）：

- タイプ定義・分類の境界を変えた／新タイプを足した
- 命名規則・テンプレート・運用ルールを新設・変更した
- 「この項目はこのタイプに入れない」等、却下・見送りの判断をした

## 完了チェックリスト

提出前に確認する：

- [ ] **受け入れ要件を満たす**ことを確認した（対象外なら reject し `docs/out-of-scope.md` に記録して終了）
- [ ] 詳細ファイルが正しいタイプのディレクトリに、英語スラッグ名で作られている
- [ ] フロントマター（`title`/`slug`/`type`/`aliases`/`description`/`timestamp`/`tags`）が埋まり、`type` 値がタイプと一致（`node scripts/lint-okf-at` が通る）
- [ ] **命名**が一般的・確立した呼称になっている（特定出典の造語に寄っていない／英語名が別分野の同名概念と衝突しない）
- [ ] **出典が単一に偏っていない**（複数・国際的な裏づけを確認し、特定著者の整理はそう明示した）
- [ ] 命名・採否を**内部の収まり（既存項目との対比の良さ）で決めていない**（判断軸は概念自体の一般性・正確さ・出典）
- [ ] 本文 H1 が「日本語タイトル（English Name）」で、全節が埋まっている
- [ ] `似ている用語との違い` を検討した（混同しやすい概念があれば違いを書き、無ければ「特になし」と明記）
- [ ] `図解` の要否を判断した（構造が図的なら mermaid で1図。該当しなければ節ごと省略）
- [ ] **トップ README** の統合テーブルに、正しいジャンルラベル付きで1行追加した（★忘れやすい）
- [ ] `node scripts/gen-okf-index` で **index.md を再生成**した（CI が最新かを検査する）
- [ ] ジャンル別 README に一覧表を書いていない（一覧は `index.md` に委ねる）／トップ README のリンクの相対パスが正しい
- [ ] 関連項目と `[スラッグ](相対パス)` の Markdown リンクで相互リンクした（wikilink `[[スラッグ]]` を使っていない）
- [ ] ルール変更を伴うなら ADR を起こした（不要なら不要と判断した）
