# 関連セクションのリンクを Markdown リンクに統一する

## ステータス
Accepted（採用）

## 背景・課題
詳細ファイルの `関連`・`似ている用語との違い` 節では、関連項目を wikilink（`[[スラッグ]]`）で指していた。
これは Obsidian 等のツール向けの記法であり、GitHub や一般の Markdown レンダラではリンクにならず、プレーンテキストのまま表示される。
このリポジトリの閲覧は GitHub が主であるため、関連項目へ辿れない状態になっていた（[Issue #37](https://github.com/snamiki1212/awesome-thinking/issues/37)）。
また、OKF-AT のリンク規約は「Concept 間リンクは Markdown リンクで書く」と定める一方で、この 2 節だけ wikilink を例外として許容しており、規約の中でも扱いが割れていた。

## 決定
wikilink の例外を廃止し、リポジトリ内の Concept 間リンクはすべて参照元ファイルからの相対パスの Markdown リンク（`[スラッグ](相対パス)`）で書く。

- 既存の詳細ファイルの wikilink を Markdown リンクに一括変換した。
- OKF-AT 規約（[docs/okf-at/README.md](../../docs/okf-at/README.md)）から wikilink の許容を削除し、準拠チェックリストに項目を追加した。
- `lint-okf-at` に wikilink の検出を追加し、CI で再発を防ぐ。規約を説明するための記載（コードスパン・コードブロック内）は検出の対象外とする。
- `add-thinking-concept` スキルのテンプレート・手順・チェックリストを Markdown リンクに更新した。

## 理由
- 一覧から詳細へ、詳細から関連項目へと辿れることが、このリポジトリの読み方の前提である。リンクが GitHub 上で機能しない記法は、この前提を壊す。
- OKF はリンクを Markdown リンクで書くと定めている。例外をなくすことで、OKF-AT の規約が一貫する。

## 検討した代替案
- **wikilink を維持し、Obsidian での閲覧を前提にする**: 閲覧の主戦場は GitHub であり、リンクが機能しない状態が続くため却下。
- **wikilink と Markdown リンクを併記する**: 同じ関係を二重に書くことになり、更新漏れの温床になるため却下。
- **リンクテキストを日本語名にする（`[日本語名](相対パス)`）**: 既存の「スラッグ＋（日本語名）」の書式を崩さず機械的に変換できる `[スラッグ](相対パス)` を採用した。日本語名は従来どおり括弧で併記する。

## 影響
- 今後の項目追加では、関連項目を最初から Markdown リンクで書く（スキルのテンプレートは更新済み）。
- 本文に wikilink が混入すると CI（`lint-okf-at`）が fail する。
- [ADR: 詳細ファイルに「似ている用語との違い」節を追加する](../20260604-similar-terms-section/README.md) が定めた節構成は維持し、同 ADR 中のリンク表記（`[[スラッグ]]` でリンクする）だけを本 ADR で置き換える。
