# 収録の受け入れ要件を定め、対象外は却下リストに記録する

## ステータス
Accepted（採用）

## 背景・課題
これまでの項目追加フロー（`docs/taxonomy.md` / スキル `add-thinking-concept`）は、
「4タイプのどれに入れるか」の**分類**から始まっていた。
その手前の「**そもそも思考ツールとして収録対象か**」という受け入れ判定が明文化されておらず、
- ドメインの専門知識・事実、単なる用語、製品の固有名、既存項目の別名などが
  紛れ込んでも弾く根拠がない
- 仮に弾いても判断が記録されず、同じ提案が繰り返し持ち込まれて再審査になる
という穴があった。

## 決定
1. **受け入れ要件（admission gate）を `docs/taxonomy.md` に明文化**する。
   4タイプ判定の前段として、「収録対象（思考ツール）の条件」と「収録対象外（reject）の例」を定義する。
2. **対象外と判断したものは reject し、`docs/out-of-scope.md`（却下リスト）に1行記録**する。
   詳細ファイル・インデックスは作らない。
3. スキル `add-thinking-concept` に **手順0「収録可否を判定する」** を追加し、
   reject 時の伝え方・記録先・打ち切りを手順化する。

## 理由
- **品質と一貫性**: 「思考ツール」という収録範囲を明示することで、雑多な知識の混入を防ぐ。
- **再審査の防止と判例化**: 却下を1か所に残すことで、同じ提案の蒸し返しを防ぎ、境界判定の判例集になる。
- **軽量さ**: ルーチンの却下までフル ADR にすると重い。却下リスト（追記1行）に逃がし、
  ADR は「受け入れ要件そのものを変える／新タイプを足す」判断にだけ使う。
- 正本の所在を概念ごとに1つに保つ方針（[procedure-in-skill](../20260604-procedure-in-skill/README.md)）と整合する。
  受け入れ要件の定義は `taxonomy.md`、判定の手順はスキル、却下の記録は `out-of-scope.md`。

## 検討した代替案
- **却下のたびに ADR を起こす**: 経緯は最も厳密に残るが、項目1件ごとに ADR は重く運用が回らないため却下。
  CLAUDE.md の「却下・NG も記録する」要請は、軽量な却下リスト＋要件変更時の ADR で満たす。
- **記録せず破棄する**: 最も軽量だが、同じ提案の蒸し返しと判断のブレを招くため却下。
- **却下と保留（borderline）を分けて二重管理する**: 当面の量では過剰。
  境界事例は「最も主要な用途で4タイプに寄せて収録」する既存方針で吸収できるため見送り。

## 影響
- `docs/taxonomy.md` に「収録の前提（受け入れ要件）」節を追加。
- `docs/out-of-scope.md` を新設（却下リスト）。
- スキル `add-thinking-concept` に手順0と、完了チェックリストの受け入れ確認項目を追加。
- 今後、項目追加の依頼を受けたらまず受け入れ要件で判定し、対象外なら理由を伝えて却下リストに記録する運用とする。
