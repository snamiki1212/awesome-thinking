# 手書き文書には規則だけを書き、実体の列挙は index.md に委ねる

## ステータス

Accepted（採用）

## 背景・課題

[README の一覧列挙を index.md に委ねる ADR](../20260713-readme-delegates-listing-to-index/README.md) で、README.md の一覧表は撤去した。
しかし CLAUDE.md とトップ README の「ディレクトリ構成」節には、`docs/taxonomy.md` や `scripts/lint-okf-at/` といった個別ファイル・ツールの列挙が残っていた。
この列挙も実体のインベントリであり、ディレクトリやツールが増えるたびに手で直す必要がある。
実際に index.md 生成ツールの導入時、CLAUDE.md のツリーへ `scripts/gen-okf-index/` などを手で追記しており、放置すれば実体と食い違っていく。

一方で、「ディレクトリ構成」節を丸ごと消すこともできない。
`<slug>.md` や `yyyymmdd-<name>/` のような、まだ存在しないファイルの配置パターンは、実体の目次である `index.md` では表現できないため。

## 決定

CLAUDE.md・トップ README のような手書き文書には、次の書き分けを適用する。

- **書いてよいもの（規則）**: 配置のパターン（`<slug>.md`、`yyyymmdd-<name>/` など）、各ディレクトリ・ファイルの役割、および決定で閉じた集合の列挙（4タイプのディレクトリ名など。増減には必ず ADR を伴うため陳腐化しない）
- **書かないもの（インベントリ）**: 増減しうる実体の列挙（`docs/` や `scripts/` 配下の個別ファイル・ツールなど）。ディレクトリ単位の役割コメントまでにとどめ、網羅は `index.md` へのリンクで委ねる

## 理由

- インベントリは実体が増減するたびに陳腐化し、手書き文書との食い違いを CI は検査できない。`index.md` は CI が鮮度を保証する
- 規則と閉じた集合は、変更に ADR を伴うため更新頻度が低く、手書きでも食い違いが起きにくい。「陳腐化するものは生成物へ、しないものは手書きへ」という置き場所の使い分けになる

## 検討した代替案

- **「ディレクトリ構成」節を丸ごと削除して index.md へのリンクだけにする**: 最も単純だが、配置パターンという規則の置き場が失われる。`index.md` は存在するファイルしか列挙できず、「新しい項目をどこにどう置くか」を伝えられない
- **CLAUDE.md のツリーも自動生成する**: 規則（未存在のパターン）と実体が混ざったツリーは機械生成できない。手書き本文と生成部分の混在は、README の一覧で却下した構図と同じ
- **現状維持（ツリーに実体も列挙し続ける）**: ツール・文書を追加するたびに CLAUDE.md とトップ README の更新が必要で、更新漏れを検出する仕組みがない

## 影響

- CLAUDE.md・トップ README の「ディレクトリ構成」から `docs/` `scripts/` 配下の個別列挙を落とし、ディレクトリ単位の役割コメントに置き換える。両節に「網羅は index.md を参照」の案内を添える
- 今後ツールや運用ドキュメントを追加しても、両ファイルの更新は原則不要になる（新しい「種類」の置き場を作ったときだけ、規則として追記する）
- CLAUDE.md 内の用語テーブル（3階層・4タイプ）などの閉じた列挙は、この決定の対象外としてそのまま維持する
