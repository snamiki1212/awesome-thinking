# 詳細ファイルに任意の「図解」節を設け、mermaid で図示する

## ステータス
Accepted（採用）

## 背景・課題
木構造・循環・4象限のように、概念の構造そのものが図的な項目が、文章と箇条書きだけで説明されていた（[issue #38](https://github.com/snamiki1212/awesome-thinking/issues/38)）。
こうした項目は、図が1つあるほうが文章より速く構造が伝わる。
一方で、すべての項目に図が要るわけではない。
構えや姿勢を述べる項目に形式的な図を強いると、手順の言い換えにしかならない図が増え、かえってノイズになる。

## 決定
詳細ファイルのテンプレートに、**任意の節 `## 図解`** を追加する。

- 置くのは、概念の構造そのものが図的な項目に限る。目安は、木構造・循環（サイクル）・2軸の行列・フィードバックループ・抽象と具体の往復など。
- 置き場所は `## 定義` の直後で固定する。
- 図は mermaid で記述し、1項目につき1つを基本とする。図の前に、何を図示したかを述べる1〜2文を添える。
- 該当しない項目では節ごと省略する。全項目必須にはしない。

## 理由
- mermaid はテキストとして差分レビューでき、GitHub 上でそのままレンダリングされる。リポジトリ内でも README と `docs/taxonomy.md` で採用済みで、道具が増えない。
- 任意にするのは、図が本質でない項目に図を強いると、本文の言い換えにしかならない図が増えるため。
- 置き場所を `定義` の直後に固定するのは、項目ごとに図の位置がばらつくと読者が探すことになるため。

## 検討した代替案
- **全項目に図解を必須とする**: 構え・姿勢を述べる項目（思考法の多く）では図が本文の言い換えになる。却下。
- **節を増やさず、既存の節（定義・使い方など）の中に図を埋め込む**: 図の位置が項目ごとにばらつき、テンプレートからも図の存在が読み取れない。却下。
- **画像ファイル（PNG・SVG）で図示する**: 差分レビューができず、作図手段の管理も必要になる。却下。

## 影響
- 項目追加スキル `add-thinking-concept` のテンプレートと完了チェックリストに、任意節「図解」を追記する。
- 初期適用として、構造が図的な既存7項目（ロジックツリー・なぜなぜ分析・PDCA・SWOT・システム思考・デザイン思考・抽象化と具体化）に図解節を追加する。
- 今後の新規項目は、構造が図的かどうかを判断したうえで図解節の有無を決める。
