# 思考ツールに不変の通し番号 ID を付ける

## ステータス

Accepted（採用）

## 背景・課題

項目は日本語名・英語スラッグ・タイプで参照できるが、いずれも名称変更・分類変更・配置変更の影響を受ける。
変更履歴で「どの項目が追加・変更されたか」を長期に追うには、これらと独立した安定した識別子が必要である。

## 決定

思考ツールの詳細ファイルに、`AT-0001` から始まる通し番号 ID を付ける。
ID は一度割り当てたら変更・再利用しない。

数字部は 4 桁以上の十進連番とする。
`AT-9999` の次は `AT-10000` とし、桁数の増加のために ID 規則を変更しない。

ID の正典は各詳細ファイルの frontmatter とする。
トップ `README.md` と自動生成 `index.md` は ID を表示し、linter が対象ファイルの欠落・形式・重複を検査する。
詳しい運用規則は [docs/thinking-ids.md](../../docs/thinking-ids.md) に定める。

次番号は、現在のブランチにある frontmatter の最大値から求める。
次番号だけを保存するキャッシュファイルは置かない。
並行する変更で同じ番号が選ばれたときは、rebase 後に番号を振り直し、linter で重複がないことを確認する。

## 理由

通し番号は種類や名前に依存しないため、分類を見直しても参照を保てる。
短い ID を一覧に表示すれば、変更履歴と Issue で項目を取り違えずに指せる。

frontmatter を正典にすれば、各一覧は既存の生成処理から作れ、同じ ID を複数箇所で管理しない。

次番号のキャッシュは正典と別に更新が必要になる。
更新漏れやマージ競合で frontmatter と食い違うため、対象ファイルを走査して求める方が信頼できる。

採番は新しい項目を追加するときだけ行う。
走査対象も思考ツールの詳細 Markdown にある `id` の frontmatter に限られるため、通常の追加作業で待ち時間になるほどのコストは見込まない。
走査が実際に運用上の問題になったときだけ、計測結果を基にキャッシュの導入を再検討する。

4 桁で開始すれば一覧で読みやすく、4 桁以上を許せば 9,999 件を超えても ID を変えずに継続できる。

## 検討した代替案

- **スラッグだけを使う**：既存の URL と一致する利点はあるが、名称変更やディレクトリ変更に追随する。長期の参照先としては安定しないため却下した。
- **タイプ別の連番を使う**：分類が一目で分かるが、タイプを移すと ID の意味が変わる。分類と識別を分離するため却下した。
- **UUID を使う**：機械的な一意性は得られるが、変更履歴や会話で人が扱いにくい。項目数と運用規模には短い連番で足りるため却下した。
- **数字部を常に 4 桁に固定する**：当面の件数には足りるが、10,000 件目でルール変更が必要になる。初期表示の読みやすさを保ちながら継続できる 4 桁以上の形式を採る。
- **最初から 5 桁以上に固定する**：上限を先送りするだけで、桁数を選ぶ根拠がない。必要になったときだけ桁数を増やせる形式を採る。
- **次番号のキャッシュファイルを置く**：次番号は読み取りやすくなるが、frontmatter と二重管理になる。採番は項目追加時だけであり、走査対象も詳細ファイルの `id` に限られる。走査コストより不整合を防ぐ価値が大きいため、正典を増やさない方を採る。
- **一覧だけに番号を振る**：詳細ファイルを正典にできず、一覧の再生成時に対応を失う。frontmatter に ID を持たせる方を採る。

## 影響

- 既存の思考ツールすべてに ID を初期割り当てする。
- 新しい思考ツールを追加するときは、未使用の次番号を frontmatter に入れる。
- 変更履歴・ADR・Issue では、必要に応じて ID で項目を参照できる。
